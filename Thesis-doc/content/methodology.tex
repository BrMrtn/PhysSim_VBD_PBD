\section{Integration architecture}
\label{sec:Architecture}
%----------------------------------------------------------------------------
\begin{itemize}
    \item Both solvers share a common scene scaffolding.
    \item Simulations are driven by the Update method in Unity's MonoBehaviour classes.
\end{itemize}
The scaffolding handles:
\begin{itemize}
    \item mesh and spring topology construction at scene load,
    \item recording of statistics such as frametime and memory,
    \item rendering, which runs on a separate thread from the simulator.
\end{itemize}
About the frame rate:
\begin{itemize}
    \item The Update method in Unity Editor gets called as often as the environment allows. If a step takes more compute time than the frame time, the next frame is delayed until it finishes.
    \item So taking these into account, the Update method virtually gives us a variable frame rate that we can use to measure computation time.
    \item For the sake of keeping the experiments deterministic and easier to control, I used a fixed simulation delta time of $dt = 1/30$~s. This also counts as a relatively large time step, so it is a good test of the solvers' stability (and lets substep count be more meaningful in \autopageref{ssec:ErrorVsMs}).
\end{itemize}
\paragraph{Overhead of the shared scaffolding.}
\begin{itemize}
    \item The scaffolding adds a small, practically constant per-frame overhead (mostly Unity-specific calls and rendering).
    \item I measure FPS with Unity Profiler, which calculates based on the actual frames being drawn, so the value includes the overhead, but because both solvers pay this overhead identically, it does not bias the comparison.
\end{itemize}

\section{How the comparison is kept fair}
\label{sec:Fairness}
%----------------------------------------------------------------------------
\begin{itemize}
    \item I adopt four explicit fairness controls.
\end{itemize}

\subsection{Stiffness $\leftrightarrow$ compliance mapping}
\label{ssec:StiffCompl}
\begin{itemize}
    \item VBD takes a stiffness $k$ directly.
    \item XPBD takes a compliance $\alpha = 1/k$.
    \item To match materials I set:
\end{itemize}
\begin{equation}
    \alpha_{\mathrm{XPBD}} \;=\; 1/k_{\mathrm{VBD}},
\end{equation}
\begin{itemize}
    \item Mapping verified in \autoref{ssec:HarmonicOscillator}: a single spring with known analytic parameters results in the same oscillation frequency and decay rate in both solvers.
\end{itemize}

\subsection{Matched compute budget --- and the substep/iteration caveat}
\label{ssec:ComputeBudget}
\begin{itemize}
    \item The natural unit of work differs: XPBD's primary lever is substep count, VBD's is inner iteration count.
    \item \textbf{A substep is not an iteration.} A substep recomputes the inertial target $\mathbf{y}$ and solves a fresh constraint system; an iteration refines the same step.
    \item Because of this asymmetry, I try to compare the two methods based on \emph{wall-clock time} throughout \autoref{chap:Experiments}.
    \item A measure that creates bias: VBD uses maximum iteration count, XPBD uses fixed iteration count. Both could implement either, but these decisions are inherently parts of each method's design (even if I consider their future optimisations), so I keep them as they are.
\end{itemize}

\subsection{Matched gravity, $dt$, and damping}
\label{ssec:MatchedDtGravity}
For all experiments unless otherwise noted:
\begin{itemize}
    \item gravity $\mathbf{g} = (0, -9.81, 0)$~m/s$^{2}$,
    \item simulation time step $dt = 1/30$~s,
    \item Rayleigh damping is not used in either method, because the difference in solver structure creates different results with the same parameters.
\end{itemize}

\subsection{Handling VBD's accelerations for the core comparison}
\label{ssec:DisableChebyshev}
\begin{itemize}
    \item The 2024 VBD paper~\cite{Chen2024} reports significant speedups from Chebyshev acceleration and adaptive initialisation.
    \item Adaptive init is VBD's own inertial prediction, that is also present in XPBD (it guesses with explicit euler), so I keep it enabled.
    \item Chebyshev-acceleration is also part of VBD, but it confounds an apples-to-apples comparison, as XPBD could also implement the same method as shown in \cite{10.1145/2816795.2818063}.
\end{itemize}

\section{Self-collision handling}
\label{sec:SelfCollision}
%----------------------------------------------------------------------------
The two solvers implement self-collision differently, by necessity:
\begin{itemize}
    \item \emph{XPBD self-collision} uses non-penetration constraints projected each iteration --- contact enforced as a position constraint.
    \item \emph{VBD self-collision} uses an elastic penalty potential added to $E$, contributing a gradient and Hessian to the affected vertex blocks.
\end{itemize}
\begin{itemize}
    \item This is a model difference, not a solver difference: a penalty model \emph{could} be implemented in XPBD and a projection model \emph{could} be implemented in VBD, but the natural, most solver-fitting form differs.
    \item Thus the self-collision comparison experiment in \autoref{ssec:SelfCollisionDrop} is purely qualitative.
\end{itemize}

\section{Parallelisation (forward-looking)}
\label{sec:Parallelisation}
\label{ssec:Parallelisation}
%----------------------------------------------------------------------------
\begin{itemize}
    \item All measurements in this thesis are run single-threaded on the CPU.
    \item Noted here, and revisited in \autoref{sec:ThreatsValidity}, the two solvers have different parallelisation profiles:
    \begin{itemize}
        \item \emph{VBD} parallelises naturally over vertex blocks via graph colouring: vertices sharing no constraint can be updated independently within a colour class. Well-suited to GPU execution.
        \item \emph{XPBD} can be parallelised via constraint-graph colouring, but the colouring is finer-grained (per constraint -that we usually have much more of than vertices-, not per vertex) and the dependency structure is less regular.
    \end{itemize}
    \item The expected result --- parallelisation favours VBD more than XPBD --- is plausible but unverified by my experiments.
    \item This stands as a threat to external validity for any claim about which solver is ``faster overall''.
\end{itemize}
